## Purpose Visibility Investigation (OpenUSD Bug)

### Issue & PR
- Issue: https://github.com/jensjebens/OpenUSD/issues/7
- Draft PR: https://github.com/jensjebens/OpenUSD/pull/8
- Branch: `fix/purpose-visibility-in-hydra` on `jensjebens/OpenUSD`

### Root Cause
`UsdImagingDataSourcePrim::Get()` in `dataSourcePrim.cpp:772-784` only reads base `visibility` attr. Never touches VisibilityAPI purpose-specific attrs. `HdFlattenedVisibilityDataSourceProvider` only flattens a single boolean. Confirmed bug via usdrecord rendering: `renderVisibility=invisible` and `proxyVisibility=invisible` have no effect.

### Tests
3 failing (the bug), 9 passing (baseline) in `tests/test_purpose_visibility_hydra.py`

### Design Decision: Extend core HdVisibilitySchema (Option A)
- VisibilityAPI is Pixar's own schema — Hydra should match
- Separate schema in usdImaging rejected (cross-layer deps, splits visibility logic)
- Goal: help Pixar plug the gap in their spec, not sneak something past them

### Fix Plan (4 layers)
1. `pxr/imaging/hd/hdSchemaDefs.py` — add purpose visibility booleans
2. `pxr/imaging/hd/flattenedVisibilityDataSourceProvider.cpp` — flatten per-purpose
3. `pxr/usdImaging/usdImaging/dataSourcePrim.cpp` — read VisibilityAPI attrs locally
4. Storm/hdPrman purpose filtering — consume flattened purpose visibility

### Collaboration
- Units reviewed PR: agreed on fix location, raised schema extension concern, resolved in favour of core schema
- Newton: acknowledged, not currently blocking for physics work
- Verified all Hydra 2.0 delegates (Storm, hdPrman, core adapter) read same `HdVisibilitySchema::GetVisibility()` — fix is renderer-agnostic
